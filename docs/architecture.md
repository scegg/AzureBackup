# 架构

> 状态:设计草案。与 [需求](requirements.md) 和 [格式规范](format-spec.md) 配套阅读。

## 1. 工程布局与「独立还原」的解释

需求要求还原是**独立的可执行/镜像**。本仓库采用 **monorepo + 共享 Core 库**:

```
AzureBackup.Core      共享库:格式模型、加密、压缩、打包、Blob 访问
   ├── AzureBackup.Backup    →  azbackup   (备份镜像 docker/backup.Dockerfile)
   └── AzureBackup.Restore   →  azrestore  (还原镜像 docker/restore.Dockerfile)
```

- 两者是**独立的可执行与独立镜像**,满足「还原不在备份运行体内」。
- 共享 `Core` 是为了**保证写入端与读取端的格式(结构/加密/分卷)永不漂移**——这是数据安全的硬要求。
- 若你坚持物理上两个仓库,可后续把 `Core` 抽成 NuGet/子模块;当前以一致性和迭代速度优先。

## 2. 备份数据流(azbackup)

```
                        有界队列 + 背压(保证上传端始终有数据)
 ┌────────┐   ┌────────┐   ┌─────────┐   ┌─────────┐   ┌────────┐   ┌──────────────┐
 │ Scan   │──▶│ Diff   │──▶│ Pack    │──▶│ Compress│──▶│Encrypt │──▶│ Upload        │
 │ 遍历源  │   │ 变更判定│   │ 分组/分卷│   │ 7z ultra│   │AES-GCM │   │ Archive 层     │
 └────────┘   └────────┘   └─────────┘   └─────────┘   └────────┘   └──────┬───────┘
   挂载卷        hash 比对                跳过不可压类型                        │
                                                                            ▼
                                                            ┌────────────────────────────────┐
                                                            │ 写结构 root/struct 分片(Hot,加密)│
                                                            │ → 保留策略 → (可选)GC/远端verify  │
                                                            └────────────────────────────────┘
```

### 2.1 Scan
- 递归遍历 `/backup/source`。
- 加载排除文件(env 指定路径),按 gitignore 语义过滤(支持 `!` 取反)。

### 2.2 Diff(变更检测,mtime 为闸门)
- 读取上一快照的文件清单(path → size, mtime, hash)。
- 规则:**mtime 未变 → 跳过(不读内容、不算 hash)**;**mtime 变 → 重算 hash**,hash 同→未变(仅更新 mtime),hash 异→变更;`FORCE_HASH` 可强制全量。
- 变更/新增的文件交给 Pack;未变文件在新快照的 tree 中**按 hash 引用既有内容**(物理位置由全局索引解析)。

### 2.3 Pack(打包/分组/分卷)
- 同目录小文件合并进一个 pack(参数 `GROUP_FILE_MAX` / `PACK_TARGET_SIZE` / `PACK_MAX_FILES`);大文件单独成 pack。
- pack 内每段内容是一个条目(按 hash 寻址,记 entry/偏移);文件路径与元数据存于 tree,`hash→pack/entry` 存于全局索引。
- pack 字节流经压缩+加密后,**按 ~100MB 切分为多个卷**,每卷一个 blob。

> ⚠️ **分组与「不压缩」的交互(实现须谨慎)**:一个 pack 采用**统一的压缩参数**(整 pack 要么压缩、要么 store),因此分组必须**先按压缩策略分区**:
> - 同一目录的小文件即使要合并,**压缩的文件与不压缩(store)的文件不能放进同一个 pack**;
> - 结果是一个目录的小文件可能产出 **≥2 个 pack**(一个压缩包 + 一个存储包);
> - 分组算法:`同分组键(如同目录)` × `压缩策略(compress/store)` 作为联合分区键,再在每个分区内按大小阈值合并。

### 2.4 Compress
- 7-Zip(LZMA2)ultra;命中「不可压类型」清单的文件以存储模式打包。

### 2.4b Spool(本地暂存队列 + 背压)
- **两段存储**:卷先产到**压缩工作区(work dir)**;**仅当整个 pack 的 compress+encrypt 成功**,其**全部卷才原子性一次性纳入 spool**(`AZBACKUP_SPOOL_DIR`)等上传。压缩中途报错 → work dir 残卷全部作废丢弃。
  - 原因:压缩档收尾前非有效档(7z 头在最后),早产的卷不可独立使用;不能逐卷流入 spool。
- 背压双上限(任一触发):卷数 `SPOOL_MAX_FILES`(默认 500)、总字节 `SPOOL_MAX_BYTES`(默认 2G)。**水位只算 spool 中已就绪的卷**。
- **门控在「开始下一个 pack 前」**:水位未达限制 → 不论本 pack 大小,整包处理并全部纳入(故 **spool 可能超限**);水位已达 → 等待,不开新 pack。
- ⚠️ **压缩串行(producer 单线程)** → 超限上界 = 一个 pack,不会并发纳入失控。**上传并发**(`UPLOAD_CONCURRENCY`,默认 5)。
- 卷**上传成功即逐个删临时文件**,水位下降到限制以下才恢复压缩。

### 2.5 Encrypt
- AES-256-GCM。每个 pack 用随机内容密钥,内容密钥由主密钥(密码 Argon2id 派生)包裹后存于**索引分片**(`struct/*`)。详见 [格式规范](format-spec.md)。
- blob 名不透明,不泄露任何路径/文件名信息。

### 2.6 Upload
- 数据卷(`packs/*`)写入 **Archive 层**;结构文件(`root/refs/struct/config`)写入 **Hot 层**。
- 已存在(内容寻址命中)的 pack 不重复上传。
- **可取消上传**:逐卷、卷内逐 block 上传,各边界处校验源文件 mtime;变化即中止剩余卷/块(配合 §3.1 复查重做),把 Archive 早删降到最低。

## 3. 一致性、锁、中断与重试

- **分布式锁**:对 Hot 层一个 `lock` blob 取**租约(lease)**,运行时**自动续租**。取不到→说明上次未完成→**跳过本次**。
- **崩溃自恢复**:进程崩溃后**租约自动过期(≤~1 分钟)**,下次运行即可获锁——**无需手动解锁**。
- **按 pack 增量提交(替代快照级一次性提交)**:
  - 一个 pack 的**全部卷上传完成**后,立即把该 pack 登记进**索引分片**(`struct/*`,Hot)。已完成 pack **跨中断保留**,下次/续跑凭内容寻址直接复用,**不重传**。
  - 某 pack 的卷**未传完**(多卷只传了一部分)→ 这些半成品卷是**孤儿**,启动时/提交前清理。
  - **快照根对象(`root/<id>`)在所有所需 pack 就绪后发布**;发布成功才算该次备份成功。未发布的 run 其新 pack 仍可被后续 run 复用(内容寻址),不浪费。
- **失败退避重试**:`5s → 30s → 90s → 300s → 之后每 300s`,**总时长上限 2 小时**,超时则该上传操作失败并记入报告。
- **断点续传**:依赖 Azure **Block Blob 分块暂存**——大卷拆 block 暂存(stage)后一次性 commit;中断后未 commit 的 block 7 天内有效,可仅补传缺失 block。不可用时整卷重传。

### 3.1 源文件备份期间被修改的处理

- **首选(推荐用户侧)**:挂载**时间点快照**(LVM/btrfs/ZFS/云盘快照)的只读视图,而非活动卷 → 跨文件一致。
- **内置兜底——复查 + 重做,尽早拦截**:复查 mtime 分三层检查点,越早越好以避免无谓上传与 Archive 早删:
  - **① 上传前**(压缩/加密后即将上传):变则直接重做,零上传浪费。
  - **② 上传中**(逐卷、卷内逐 block):发现变化即**中止剩余上传**、丢弃未提交卷/块,只删已提交卷。
  - **③ 上传后兜底**:整 pack 提交后再查,变则删除已上传卷 blob + 重做(⚠️ 早删计费)。
  - mtime 变 → 重算 hash,hash 变才重做;**重做以 pack 为单位**;限 `AZBACKUP_VOLATILE_FILE_MAX_REPACK`(默认 3),仍变则保留最后一次并入报告。
  - 🔧 设计含义:**上传阶段需可取消、且能感知源文件 mtime**(见 §2.6)。
- **可选**:`pre/post` 钩子用于应用静默(DB flush/lock 等)。
- ⚠️ 仅快照能避免**单文件撕裂读**;兜底方案只保证每文件自洽,不保证跨文件时间点。

**部署耦合(重要)**:容器在启动时一次性挂载卷,**无法在每次任务前为自己更换快照**。因此:
- **常驻容器 + 内置 cron** → 仅能用「读后校验」兜底。
- **快照一致性** → 必须 **一次性容器 + 外部调度**(宿主 cron / K8s CronJob),每次运行前由编排方**创建并挂载新快照**,跑完销毁。
- 折中(不推荐):特权容器自行 `lvcreate`+`mount`,牺牲可移植性与隔离。

## 4. 保留、GC 与压实

> 基于 [格式规范](format-spec.md) 的 **hash 间接索引**:快照/tree 只按内容 hash 引用,`hash→物理位置` 由全局索引维护。下述操作**只改索引,不动已写快照**。

- 保留:按 env 的「最近 X 个 / 最近 Y 天」标记应删除的快照。
- **GC(引用计数)**:`live = 所有保留快照 tree 可达的 hash 集合`;不在其中的 hash 从索引剔除;`liveCount=0` 的包(全部卷)删除。
- **压实(`AZBACKUP_PACK_COMPACTION`,默认 30%)**:某包 `死重比例 = 1 - liveCount/members ≥ 阈值` 时,取其存活成员内容**重打成新包**、更新索引 `hash→新包`、删旧包回收空间。
  - 取存活内容:**优先从源**(仍在挂载卷且未变);**仅存于历史快照的成员需解冻 Archive**。
  - ⚠️ 压实会**解冻 + 重写 + 早删计费**——这是默认开启压实的代价(已接受"费用没所谓");`off` 或提高阈值可减少。
- ⚠️ Archive 早删除计费:删除/重写早于 180 天的卷会产生早删费用(已接受)。
- **执行时机可配**:备份后顺带执行,或独立 cron。

### 4.0 复杂交叉替换的处理

老包经多次变更后,各文件的不同版本会**交叉散落**在多个包、被不同快照子集引用。靠 **hash 间接索引**根治:
- tree 按 `hash` 引用 → 同一文件的每个版本是**不同 hash、各自索引条目**,天然无歧义。
- 压实/重定位**只改索引 `byHash` 条目**,**不改任何已写快照/tree**(它们不可变)。
- 链式压实(新包以后又被压实)同理,只是再次更新索引。
- 还原解析:`文件 hash →(索引)→ 当前 pack/卷`,永远指向内容**当前**所在处。

### 4.1 远端 verify(备份工程内的维护任务)

- 与 GC 同属 azbackup 的可调度维护任务:列举各保留快照引用的全部卷 blob,**仅校验存在性**(不下载、不触 Archive 读取)。
- **调度方式与 GC 一致**(`AZBACKUP_VERIFY_MODE`):`after-backup` / `cron` / `both` / `off`;`cron` 时用 `AZBACKUP_VERIFY_CRON`。
- 发现缺失卷/孤儿 → 记入报告并(按通知配置)触发 webhook。
- **本地 verify(下载+hash 比对)在还原工程**,不在此处(避免备份工程读取 Archive)。

## 5. 调度

- 容器内常驻 + cron 表达式(env),到点触发一次备份;或外部 CronJob 单次运行。
- 与锁配合实现「未完成则跳过」。
- **多任务(jobs 文件)**:一个 `AZBACKUP_CRON` 触发整批;批运行器**按 jobs 顺序**逐个执行(压缩本就串行)。每 job 独立 container/仓库/`lock`/报告;某 job 失败不阻断其余,失败汇入整批报告与通知。env 作用域三分类见 [配置](configuration.md)。

## 6. 还原与校验(azrestore)

见 [还原](restore.md)。要点:加载并解密结构(`root/refs/struct`)→ 列出备份 → 选择 → 全量或按树选择 → **解冻所需 pack 的全部卷** → 下载 → 解密 → 解压 → 写入目的地(含空目录重建)。

**本地 verify 在 azrestore**:解冻 + 下载 + 解密,**逐文件比对内容与 hash**。
(远端 verify 在备份工程,见 [§4.1](#41-远端-verify备份工程内的维护任务)。)

## 7. 可观测性与通知(azbackup)

- **结构化日志** + **退出码**(见 [usage](usage.md))。
- **运行报告**:新增/未变/上传字节、pack/卷数、耗时、被修改文件、错误。可写文件 / stdout。
- **备份量统计**:本次上传字节 + 仓库累计占用(不做费用估算)。
- **通知**:有界事件(默认 error,可配 success/both)触发 webhook。
  - **Bark**:`URL + 标题 + 内容`,POST 或 GET(`https://api.day.app/<key>/<title>/<content>`)。
  - 通用 webhook:可配方法与标题/内容模板。
